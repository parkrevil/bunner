# DOCS WRITING (Global)

## 역할

- 이 문서는 레포 전체 문서(SSOT 및 일반 문서 포함)에 적용되는 **문서 작성 규율**을 정의한다.
- 특히 L3 SPEC 문서에서 규칙이 “금지 목록”으로 붕괴되는 것을 방지한다.

## 목적

- "나는 이렇게 생각한다"가 아니라, **유효/무효를 판정 가능한 규칙**만 남긴다.
- `MUST` 규칙을 중심으로 계약을 구성하고, `MUST NOT`은 `MUST`의 허점만 방어하도록 강제한다.

## 적용 범위

- 이 문서는 `docs/**` 하위의 모든 문서 변경에 동일하게 적용된다.
- 문서 종류(SSOT, SPEC, 일반 문서)에 따른 예외는 없다.

## 용어 정의

### 판정형(Decidable)

이 문서에서 "판정형"은 다음을 모두 만족하는 규칙을 의미한다.

- 입력이 명시되어 있다. (예: 특정 파일, 특정 섹션, 특정 데이터 형상)
- 위반 상태가 명시되어 있다. (예: 필수 필드 누락, 허용 집합 외 값, 정렬 규칙 위반)
- 판정이 해석/취향/비유에 의존하지 않는다.

판정형이 아닌 예시(금지):

- "적절히", "충분히", "권장" 같이 임계값/조건이 없는 표현
- 비교/관성/프레임워크 사례를 근거로 한 규칙

### Static Shape

이 문서에서 "Static Shape"는 아래 중 하나로 정의된다.

- L3 SPEC 문서: 해당 SPEC의 `2. Static Shape` 섹션
- 그 외 문서: 문서가 계약으로서 강제하는 데이터 형상(스키마/필드/허용 값)을 명시한 섹션

---

### 1) MUST First

- 문서는 먼저 `MUST`로 "무엇을 하면 유효한 구현/문서인지"를 정의해야 한다.
- `MUST`는 다음 조건을 만족해야 한다.
  - 판정형: 한 문장으로 "충족/불충족" 판정이 가능해야 한다.
  - 직관성: 은유/비교/업계 관성에 의존하지 않는다.
  - 엄격성: 모호함을 남기지 않는다. 모호하면 빌드/판정 실패 조건으로 귀결되어야 한다.

### 2) MUST NOT은 방어 규칙

- `MUST NOT`은 "나열형 금지"가 아니라, `MUST`를 우회하여 위반을 가능하게 만드는 **모든 우회 경로(구멍)**를 차단해야 한다.
- 이 문서에서 "우회 경로(구멍)"는 다음을 의미한다.
  - 문서에 정의된 `MUST`를 "형식상" 충족하는데도,
  - `MUST`가 의도한 유효성 판정 결과를 깨뜨릴 수 있는 경로(= 위반을 가능하게 만드는 경로)
- 우회 경로를 모두 차단할 수 없다면(= `MUST NOT`을 끝없이 늘려야 한다면), `MUST`가 약하거나 과도하게 넓은 것이다.
  - 이 경우 우선순위는 다음과 같다.
    1. `MUST`를 더 판정형/엄격한 형태로 재정의한다.
    2. 불필요한 자유도를 Static Shape에서 제거한다.
    3. 그래도 판정 불가능하면 즉시 중단(STOP IF UNCERTAIN)하고 재합의한다.

종료 조건(무한 확장 방지):

- 아래 중 하나라도 발생하면, `MUST NOT` 추가를 중단하고 `MUST`/Static Shape를 재작성해야 한다.
  - `MUST NOT`이 "가능한 구현 접근"을 나열하기 시작한다. (구현/알고리즘/튜토리얼로 전이)
  - `MUST NOT`이 외부 사례/비유를 근거로 하게 된다.
  - `MUST NOT`이 판정형이 아닌 표현(적절히/충분히/권장)을 포함한다.

### 3) “비지원”은 문장으로 쓰지 않는다

- 어떤 기능/필드/API를 지원하지 않음을 문장으로 선언하는 대신,
  - **형상(Static Shape)에 존재시키지 않음**으로 표현한다.
- 예외: 상위 불변식 위반(예: 런타임 스캔, reflect-metadata)처럼 정책상 즉시 중단이 필요한 금지는
  `MUST NOT`으로 1줄로만 명시할 수 있다.

### 4) 용어는 계약에서만 정의한다

- 중요한 용어는 아래 중 하나에 **명시적으로** 정의되어야 한다.
  1. 상위 정본 용어집: [docs/10_FOUNDATION/GLOSSARY.md](../10_FOUNDATION/GLOSSARY.md)
  2. 해당 계약 문서의 `Definitions` 섹션
- 정의되지 않은 용어가 판정에 필요해지면, 문서는 무효이며 작업은 즉시 중단(STOP IF UNCERTAIN)되어야 한다.
- 새 용어를 도입해야 한다면, 우선 [docs/10_FOUNDATION/GLOSSARY.md](../10_FOUNDATION/GLOSSARY.md)에 추가하고(상위 정본), 필요 시 해당 SPEC의 `Definitions`에서 재사용한다.

### 5) 전역 용어 SSOT는 GLOSSARY 하나다

- 전역 용어의 정본(SSOT)은 [docs/10_FOUNDATION/GLOSSARY.md](../10_FOUNDATION/GLOSSARY.md)만이다.
- 개별 문서(L3 SPEC 포함)는 전역 용어를 재정의해서는 안 된다.
- 위 규칙은 본 문서의 적용 범위에 의해 `docs/**` 전체에 전역 적용되므로, 개별 문서에 이를 반복 기재하지 않는다.

## 에이전트 집행 규칙 (Agent Enforcement)

- 집행 주체: 본 레포에서 작업하는 모든 AI 에이전트(Architect/Implementer/Reviewer)
- 집행 시점: 문서(`docs/**`) 변경을 적용하기 직전
- 중단 조건(STOP IF UNCERTAIN): 아래 중 하나라도 발생하면 즉시 중단한다.
  - `MUST` 또는 `MUST NOT`이 판정형이 아니다.
  - 용어가 Static Shape/Definitions/상위 정본 용어집에 정의되지 않았는데 판정에 필요하다.
  - "비지원"을 문장으로 선언해야만 계약을 표현할 수 있고, Static Shape로 표현할 수 없다.

특히 L3 SPEC 문서에서 `MUST`가 판정 불가능한 형태로 작성되면, 작업을 중단하고 수정 방향을 재합의해야 한다.
